# ServiceNow AI — Common Issues, Pitfalls & Troubleshooting

---

## Issue 1: Now Assist Producing Inaccurate or Hallucinated Summaries

**Category:** Quality / LLM Behavior
**Severity:** High — erodes agent trust and can lead to wrong actions

**The Problem:**
Incident summaries generated by Now Assist occasionally contain information that was not in the incident record — the AI appears to have "invented" details (incorrect resolution steps, wrong CI names, inaccurate timeline). Two agents have taken actions based on incorrect AI summaries.

**Why It Happens:**
LLMs can "hallucinate" — generate plausible-sounding but factually incorrect content. This is more likely when the source material is sparse (short incident with little context) or when the AI is asked to infer beyond what is in the record.

**Root Cause:**
LLM hallucination on low-context records, combined with insufficient user training on when to trust vs. verify AI output.

**Fix / Prevention:**
- Train agents that Now Assist summaries are drafts to review, not facts to act on directly
- Configure the summary prompt to instruct the LLM to only summarize information present in the record and explicitly state when information is uncertain
- Add a disclaimer banner on AI-generated summaries: "AI-generated — review before acting"
- For critical actions (P1 escalations, security incidents), require explicit agent confirmation that they reviewed the source record, not just the AI summary
- Monitor hallucination rate in AI Control Tower — if thumbs-down feedback includes "inaccurate information," investigate prompt tuning

## Issue 2: Virtual Agent Resolution Rate Below 40% — Users Bypass the Bot

**Category:** Configuration / NLU Quality
**Severity:** High — self-service investment not delivering expected deflection

**The Problem:**
Virtual Agent was deployed with high expectations for self-service deflection. Three months in, resolution rate is 38% and users are clicking "Talk to an agent" immediately rather than engaging with the bot. ITSM ticket volume has not decreased.

**Why It Happens:**
Two common root causes: (1) NLU is failing to understand user input — users get "I did not understand that" responses repeatedly and give up. (2) Topics handle the easy cases only — when the bot cannot resolve the request, the human handoff experience is poor (long wait, context not transferred).

**Root Cause:**
Insufficient NLU training utterance diversity and/or incomplete topic coverage for the most common user requests.

**Fix / Prevention:**
- Pull the VA conversation logs for the last 30 days and identify the top 20 phrases that triggered "I did not understand" — add these as training utterances for relevant topics
- Review the topic coverage: are the top 10 ITSM request types all covered by VA topics? Prioritize topics by request volume
- Improve the human handoff: ensure the full VA transcript is attached to the created ticket and the human agent is notified with full context — bad handoff experiences are a major reason users skip the bot entirely
- Set a realistic expectation internally: 40% resolution rate in month 3 is not unusual. Set a target of 60%+ at 6 months with continuous tuning

## Issue 3: Predictive Intelligence Routing Accuracy Degraded After Process Changes

**Category:** Model Drift / Maintenance
**Severity:** Medium-High — incorrect routing increasing manual reassignment work

**The Problem:**
Predictive Intelligence was routing incidents to the correct assignment group with 78% accuracy for 8 months. After an IT reorganization that changed several team names and merged some groups, accuracy dropped to 55% and the volume of manually reassigned incidents tripled.

**Why It Happens:**
The ML model was trained on historical data where "Network Team" was the correct group for a set of incidents. After reorganization, that team was renamed and merged into "Infrastructure & Network." The model still predicts the old group name, which no longer exists in the assignment group list.

**Root Cause:**
Predictive Intelligence model not retrained after organizational change — model predicts based on historical labels that are now stale.

**Fix / Prevention:**
- Retrain the Predictive Intelligence solution after any significant organizational change that affects the predicted outcome categories (team names, categories, priorities)
- Add a model maintenance trigger to the IT organizational change process: "If assignment groups are added, renamed, or merged, retrain all affected PI solutions within 2 weeks"
- Monitor prediction drift: track routing accuracy weekly in PA. A sudden drop in accuracy indicates the model needs retraining or the underlying data has changed

## Issue 4: Now Assist for Creator Generating Incorrect or Insecure Code

**Category:** Quality / Security
**Severity:** High — security vulnerabilities or broken code from AI generation

**The Problem:**
A developer used Now Assist for Creator to generate a Script Include. The generated code was syntactically correct and looked reasonable, but contained a GlideRecord query with no `addQuery()` condition — effectively querying the entire table on every invocation. This caused severe performance degradation when deployed.

**Why It Happens:**
LLMs generate code that looks correct but may miss ServiceNow-specific best practices (like always limiting GlideRecord queries) that are not always enforced by general code training data.

**Root Cause:**
AI-generated code not reviewed for platform-specific best practices before deployment.

**Fix / Prevention:**
- Establish a code review requirement for ALL AI-generated code — same standards as human-written code
- Create a Now Assist for Creator code review checklist: query limits present, no hardcoded sys_ids, proper null checks, scoped app vs. global scope appropriate
- Add security scanning to the CI/CD pipeline that catches common ServiceNow anti-patterns (unlimited GlideRecord queries, hardcoded credentials, missing ACL checks)
- Train developers: AI-generated code is a starting point, not a finished product — always review before committing

## Issue 5: AI Provider Authentication Failing — All Now Assist Skills Unavailable

**Category:** Configuration / Integration
**Severity:** Critical — all AI features down

**The Problem:**
All Now Assist skills show "Service Unavailable" errors. Incident summarization, KB suggestions, and all other AI skills are non-functional. No changes were made to the ServiceNow instance.

**Why It Happens:**
The AI Provider (Azure OpenAI in this case) had its API key rotated as part of the customer's security policy. The new key was not updated in ServiceNow's AI Provider configuration.

**Root Cause:**
AI Provider API key rotation without updating ServiceNow credential configuration.

**Fix / Prevention:**
- Add AI Provider credential rotation to the security key rotation runbook — whenever Azure OpenAI API keys are rotated, updating ServiceNow AI Provider configuration is a required step
- Set up monitoring: alert when any AI skill returns errors above a threshold (5+ errors in 10 minutes indicates a systemic issue, not individual failures)
- Consider using Azure Managed Identity for the AI Provider connection if the MID Server is an Azure VM — eliminates API key rotation issues entirely

## Issue 6: Virtual Agent Not Handing Off Correctly to Human Agents

**Category:** Configuration / Human Handoff
**Severity:** Medium-High — poor employee experience when VA cannot resolve

**The Problem:**
When Virtual Agent cannot resolve a request and the user clicks "Talk to an agent," the created ITSM ticket is blank — none of the conversation context is transferred. Agents have to ask the user to describe their issue again, creating frustration and extending resolution time.

**Why It Happens:**
The VA topic's escalation step is configured to create an incident but was not configured to pass conversation context (transcript, entities collected, user details) to the new record.

**Root Cause:**
VA human handoff step missing context transfer configuration.

**Fix / Prevention:**
- In every VA topic's escalation step, configure: transcript attachment (full conversation), entity values pre-populating incident fields (if VA collected the issue category, pre-fill the incident category), and the user's VA session context
- Test every escalation path: verify the created incident has full conversation context before go-live
- Configure the incident/case creation step to set a flag indicating the ticket originated from VA — agents can see at a glance that context is attached

## Issue 7: Predictive Intelligence Low Confidence Scores — Model Not Confident Enough to Auto-Apply

**Category:** Data Quality / Training
**Severity:** Medium — AI feature not delivering expected automation benefit

**The Problem:**
Predictive Intelligence for incident category prediction was deployed 2 months ago. Confidence scores are consistently below 60% — almost never reaching the auto-apply threshold. Every prediction is shown as a low-confidence suggestion that agents frequently ignore.

**Why It Happens:**
Two common causes: (1) Insufficient training data — fewer than 400 examples per category. (2) Inconsistent historical labeling — the same type of incident was categorized differently by different agents over time.

**Root Cause:**
Training data quality too low — either volume insufficient or labels inconsistent.

**Fix / Prevention:**
- Run a data quality audit of training records: for each category, sample 20 records and verify the category assignment is accurate. Inconsistent labeling inflates variance and reduces model confidence
- If category definitions changed over time (e.g., "Network" was split into "Network - WAN" and "Network - LAN" 18 months ago), exclude records from before the change from training data
- Increase training data volume: ensure at least 400 clean, correctly labeled examples per category
- Consolidate categories if some have very few examples — a category with 40 examples will always have low confidence

## Issue 8: AI Agents Making Incorrect Decisions — Wrong Priority Assigned

**Category:** Quality / AI Agent Decision
**Severity:** High — operational impact from incorrect autonomous decisions

**The Problem:**
An AI Agent deployed to auto-triage new incidents is assigning P1 priority to incidents that should be P3. Alerts flood the NOC and the on-call team is paged unnecessarily 3-4 times per night.

**Why It Happens:**
The AI Agent's decision logic for priority assignment was based on keyword matching in the short description — any mention of "down," "outage," or "not working" triggered P1. The agent had not been given sufficient context about the relative criticality of different CI types.

**Root Cause:**
AI Agent decision logic too simple — not incorporating CMDB CI criticality, affected user count, or business service context in the priority decision.

**Fix / Prevention:**
- Update the AI Agent's priority decision skill to incorporate CMDB data: an incident for a workstation CI should not auto-P1; an incident for a Tier 1 business service CI should
- Add a confidence threshold: the AI Agent should only auto-assign priority for cases where it has high confidence. For ambiguous cases, it should suggest rather than act
- Implement human-in-the-loop for P1 auto-assignment initially — AI suggests P1, a human confirms within 5 minutes before the P1 process activates
- Monitor AI Agent decisions in AI Control Tower: flag when AI-assigned P1 is manually downgraded by an analyst — use these as training examples for agent improvement

## Issue 9: KB Article Quality Degrading Now Assist Response Quality

**Category:** Data Quality / Knowledge Management
**Severity:** Medium — AI responses citing outdated or incorrect KB content

**The Problem:**
Now Assist is generating responses in the Employee Center chat that cite outdated HR policies. Employees are getting AI-generated answers about parental leave benefits that were changed 8 months ago. The AI is accurate relative to the KB articles — but the KB articles are wrong.

**Why It Happens:**
Now Assist grounds its responses in the Knowledge Base. If KB articles are outdated, AI responses will be outdated. AI does not know that a policy changed — it only knows what the KB says.

**Root Cause:**
KB article quality not maintained — outdated articles not retired or updated before Now Assist deployed.

**Fix / Prevention:**
- Before deploying Now Assist for any module, audit the KB for that module: review all articles older than 12 months, retire any that are no longer accurate
- Implement mandatory KB review schedules (as covered in HRSD implementation) — every article has a review date; articles not reviewed by their due date are auto-retired
- Add a "Last verified by AI" flag to KB articles: track when an article was last reviewed specifically for AI accuracy (more stringent than human readability alone)
- In AI Control Tower, configure low-confidence responses to display a disclaimer: "If this policy has changed recently, please verify with HR directly"

## Issue 10: Now Assist for Creator Code Not Scoped Correctly

**Category:** Configuration / Development
**Severity:** Medium — code generated in wrong scope causing upgrade and permission issues

**The Problem:**
Developers using Now Assist for Creator to generate Script Includes are getting global-scope code by default. ServiceNow best practice is scoped application development, but the AI keeps generating global Script Includes and Business Rules even when the developer is working in a scoped app.

**Why It Happens:**
The AI generator's context does not automatically detect the current application scope the developer is working in. It defaults to global scope patterns which is the most common pattern in its training data.

**Root Cause:**
Now Assist for Creator not inheriting or prompting for application scope context.

**Fix / Prevention:**
- Always specify scope in the prompt: "Generate a scoped Script Include for the X application that uses GlideRecord to..." — including scope context in the prompt significantly improves scoped code generation
- Add a code review checklist item: verify generated code uses scoped syntax (GlideRecord queries include scope prefix, API calls use scoped APIs, table access uses scoped permissions)
- Request ServiceNow to add automatic scope detection to Now Assist for Creator — the IDE knows which app is open, this context should be injected automatically into the generation prompt
